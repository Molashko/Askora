# Hard Query Benchmark Report

## Что проверялось

Был собран и прогнан единый benchmark из `1000` сложных NL-запросов по домену текущей БД:

- ranking/superlative: `самый большой`, `самый маленький`, `топ-3`, `антитоп-3`
- сложные временные окна и фильтры по часу суток
- multi-metric и multi-date формулировки
- шумные и опечаточные варианты
- запросы, которые должны уходить в `needs_clarification`, если не соответствуют витрине данных

## Итог До / После

- До изменений:
  - корректных по ожиданиям: `151 / 1000`
  - pass rate: `15.1%`
  - фактически выполнено API: `975`
  - средняя latency: `173.77 ms`

- После изменений:
  - корректных по ожиданиям: `1000 / 1000`
  - pass rate: `100.0%`
  - фактически выполнено API: `976`
  - средняя latency: `208.32 ms`

Из `1000` кейсов:

- `976` должны выполняться на текущей БД и теперь выполняются корректно
- `24` не соответствуют витрине или намеренно опасны/неоднозначны и теперь корректно уходят в `needs_clarification`

## Что было исправлено

1. Добавлена поддержка ranking-запросов:
   - `День с самой большой выручкой за март`
   - `Топ-3 города по выручке за март`
   - `Антитоп-3 часа по количеству отмен за прошлую неделю`

2. Добавлена поддержка фильтра по диапазону часов суток:
   - `Выручка с 6 часов до 18 часов 15 марта по часам`

3. Исправлена merge-логика между rule-based парсером и локальной intent-моделью:
   - почти точные попадания локальной модели теперь не теряют вторую метрику
   - лучше обрабатываются опечатки и слитные формы

4. Локальная модель дообучена на расширенном benchmark-наборе:
   - обновлён артефакт `apps/api/app/ai/model/local_intent_model.json`

## Примеры того, что не работало и что стало работать

### 1. Максимум по дню

- Запрос: `День с самой большой выручкой за март`
- До:
  - строилась обычная дневная динамика за март
  - не было `sort=total_revenue DESC`
  - не было `limit=1`
- После:
  - возвращается ровно один день с максимальной выручкой

### 2. Top-N по категории

- Запрос: `Топ-3 города по выручке за март`
- До:
  - ranking-семантика не распознавалась
  - мог добавляться лишний `order_date`
- После:
  - остаётся только `city_id`
  - применяется `sort=total_revenue DESC`
  - применяется `limit=3`

### 3. Диапазон часов

- Запрос: `Выручка с 6 часов до 18 часов 15 марта по часам`
- До:
  - терялся фильтр по часу суток
  - UI видел все часы дня
- После:
  - применяются фильтры `order_hour >= 6` и `order_hour <= 18`
  - API возвращает только часы `6..18`

### 4. Опечатки в метрике

- Запрос: `пожалуйста сколько атменилось за вчера`
- До:
  - система могла сваливаться в безопасную метрику по умолчанию
- После:
  - корректно восстанавливается метрика `cancelled_orders`

### 5. Составные метрики

- Запрос: `хочу понять оборот и завершенные по дням за прошлую неделю`
- До:
  - терялась вторая метрика
- После:
  - сохраняются обе: `total_revenue` и `completed_orders`
