# Анализ последних 20 пользовательских запросов

В этот файл включены последние 20 естественных формулировок пользователя до массовых synthetic benchmark-прогонов. После больших автоматических прогонов часть исходных записей в `query_history` была вытеснена более новыми тестовыми запросами, поэтому основной вывод ниже основан на повторном прогоне тех же пользовательских фраз на текущем состоянии системы.

## Короткий итог

- Проанализировано: 20 запросов.
- Сейчас корректно исполняются: 20 из 20.
- `needs_clarification`: 0.
- `failed`: 0.
- Текущий pass rate по этому пользовательскому срезу: `100%`.

## Что было в этих запросах

Срез получился показательным, потому что он покрывает сразу несколько важных классов:

- простые агрегаты по одной метрике;
- временные ряды по дням и месяцам;
- почасовую разбивку;
- multi-metric запросы;
- разрезы по статусам;
- процентные и сравнительные вопросы.

## Что система сейчас делает корректно

- Понимает процентные формулировки вроде роста выручки между периодами.
- Правильно выбирает completion-rate для запросов про процент выполненных заказов.
- Корректно обрабатывает multi-metric запросы вроде `Выручка и выполненные по дням за прошлую неделю`.
- Корректно строит status breakdown для заказов и тендеров.
- Уверенно различает `order_month`, `order_date`, `order_hour`, `order_week`, `city_id`, `order_status`, `tender_status`.

## Ограничение анализа

Для части этих запросов уже нельзя надёжно восстановить исторический `query_history`-снимок именно на момент первого пользовательского прогона, потому что позже история была частично заполнена моими benchmark-запросами. Поэтому здесь честно зафиксировано текущее, воспроизводимое поведение системы на тех же формулировках, а не реконструкция всех старых промежуточных состояний.
