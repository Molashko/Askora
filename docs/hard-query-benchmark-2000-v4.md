# Hard Query Benchmark 2000

## Итог

- Проверено: `2000` запросов.
- Корректно по ожиданиям: `1984`.
- Ошибок: `16`.
- Pass rate: `99.2%`.
- Средняя latency: `201.01 ms`.
- Фактически исполнено: `1975` запросов.
- Ожидаемо исполняемых кейсов: `1976`.
- Ожидаемо уходящих в `needs_clarification`: `24`.

## Что улучшилось

- После усиления ranking-логики, typo-recovery и stem-эвристик для метрик система уверенно держит не только `топ/антитоп`, но и `лучший/худший`, `лучшие 3/худшие 3`.
- Локальная модель выросла до `10875` entries за счёт вариаций hard-benchmark кейсов, а не только точных шаблонов.
- Benchmark-runner теперь устойчив к одиночным сбоям внутри длинного прогона: отдельный плохой кейс больше не должен обрывать весь тестовый набор.

## Остаточный хвост ошибок

Оставшиеся `16` кейсов концентрируются в трёх узких кластерах:

1. Сильно искажённые формы `день недели`, которые ещё иногда падают в `order_date` вместо `order_dow`.
2. Отдельные typo-варианты ranking-запросов по `количеству заказов` и `количеству отмен`, где иногда путается направление сортировки.
3. Один SQL/transaction edge-case, где при конфликте metric/sort alias формируется неконсистентный `ORDER BY`, после чего сессия уходит в aborted state.

## Практический вывод

Для реальных пользовательских формулировок система уже находится в хорошем состоянии. Оставшийся хвост относится к агрессивно зашумлённым synthetic mutation-кейсам, которые выходят далеко за пределы обычных опечаток.
